# 技术方案生成助手

> 上传技术规范书 → AI 按原文结构逐段扩写 → 下载完整 Word 技术方案

一款专为政企信息化投标、方案编写场景设计的 AI 辅助工具。上传采购方的技术规范书（招标文件），系统自动解析文档目录结构，调用大语言模型按原文段落逐章节扩写生成专业技术方案内容，导出为格式规范的 Word 文档。

## 整体工作流程

```
┌─────────────────────────────────────────────────────────┐
│  Step 1：上传技术规范书                                   │
│  支持格式：.docx / .pdf / .doc（自动转换）               │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────────┐
│  Step 2：文档解析（自动）                                 │
│                                                         │
│  TOC 提取策略（按优先级）：                               │
│    1. Word 内置目录（SDT 结构）→ 层级最准确              │
│    2. 手动编写目录（扫描前 80 段落）                      │
│    3. 标题样式扫描（兜底，无目录时使用）                  │
│                                                         │
│  提取结果：章节 ID + 标题 + 层级 + 原文内容              │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────────┐
│  Step 3：配置 LLM（首次使用）                             │
│  支持：OpenAI / Claude / 豆包 / Kimi / MiniMax / DeepSeek│
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────────┐
│  Step 4：设置目标字数，点击「开始生成」                   │
│                                                         │
│  生成流程：                                              │
│    ① 生成全文摘要（作为各章节的全局上下文）              │
│    ② 逐章节：有原文 → 逐段扩写；无原文 → 保留标题跳过   │
│    ③ SSE 实时流式推送，左侧目录实时显示进度              │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────────┐
│  Step 5：下载 Word 文档                                  │
│  目录结构与原文 TOC 严格一致（含跳过章节的标题）         │
└─────────────────────────────────────────────────────────┘
```

## 上传文件要求（关键）

生成质量的上限取决于输入规范书的质量。以下要求直接影响解析准确率和生成效果。

### 支持的文件格式

| 格式 | 说明 | 推荐度 |
|------|------|--------|
| `.docx` | Word 2007+ 格式，TOC 识别最准确 | ⭐⭐⭐ 首选 |
| `.pdf` | 可检索 PDF（非扫描件） | ⭐⭐ |
| `.doc` | Word 97-2003 旧格式，需服务器安装 LibreOffice 转换 | ⭐ |

> **扫描件 PDF 不支持**：若 PDF 整页为图片（无法用鼠标选中文字），系统无法提取文字，请先用 OCR 工具转为可检索 PDF 或 DOCX。

---

### DOCX 文件 — 决定解析质量的关键点

#### 关键点 1：文档必须包含目录页

系统依赖目录来精确识别章节层级和正文位置。文档应包含以下任一形式的目录：

**（推荐）Word 内置目录** — 通过 `引用 → 目录 → 自动生成目录` 插入，系统可直接读取 TOC 数据结构，层级识别最准确。

**手动编写目录** — 在文档前部有独立目录页，页面上方有"目录"二字，目录条目包含编号前缀（如 `一、` `1.` `（1）`），系统可自动扫描识别。

> 若文档完全没有目录，系统将回退到"标题样式扫描"模式，依赖 Word 的 Heading 样式识别章节，但层级准确率会下降。

---

#### 关键点 2：标题使用规范的编号格式或 Heading 样式

系统支持两种标题识别方式，**同时支持，二者均可**：

**方式 A：标准编号前缀（最通用）**

| 编号示例 | 识别层级 | 说明 |
|---------|---------|------|
| `一、总体方案` `二、系统设计` | Level 1 | 中文序数 + 顿号 |
| `第一章 项目概述` | Level 1 | 第X章格式 |
| `1. 需求分析` `2. 架构设计` | Level 2 | 阿拉伯数字 + 点 + 空格 |
| `（1）功能需求` `（2）性能需求` | Level 3 | 括号包裹数字 |
| `① 用户管理` `② 权限管理` | Level 4 | 带圈数字 |
| `1.1 系统架构` `1.2 技术选型` | Level 4 | 小数点编号 |

**方式 B：Word Heading 样式**

在 Word 中为标题段落应用「标题 1 / 2 / 3 / 4」样式，系统直接按样式层级识别，无需编号。

---

#### 关键点 3：各章节原文内容越充实，生成质量越高

系统按照"**原文有什么，就扩写什么**"的原则工作：
- 原文内容丰富（技术要求、功能描述、指标参数）→ AI 扩写有据可依，输出质量高
- 原文只有标题、无正文 → 该章节被自动跳过，仅在最终文档中保留标题
- 原文包含量化指标（如"响应时间不超过 3 秒"）→ AI 会忠实保留并展开描述

**高质量输入示例：**
```
一、平台建设目标
本项目旨在建设统一的数字化运营平台，实现各业务系统数据汇聚。
要求系统支持不少于 10 个业务系统对接，接口响应时间不超过 500ms，
平台可用性达到 99.9% 以上，支持横向扩展至 200 并发用户。
```

**低质量输入示例（仅标题，无内容）：**
```
一、平台建设目标
```
> 该章节将被自动跳过生成，最终文档中仅保留"平台建设目标"标题。

---

#### 关键点 4：文件结构注意事项

| 问题情形 | 影响 | 建议 |
|---------|------|------|
| 文档中有多个目录页 | 可能识别第一个目录 | 保留主目录，删除附录目录 |
| 标题与正文字体完全相同（无编号） | 标题可能无法识别 | 为标题应用 Heading 样式或添加编号 |
| 目录页码数字被误识别为内容 | 解析偏差 | 使用 Word 内置目录，系统可正确处理 |
| 内容中大量使用表格 | 表格内容提取为纯文本 | 正常，不影响主要结构 |
| 文件超过 20 MB | 解析缓慢或超时 | 拆分文件或压缩图片后上传 |
| 章节数超过 200 个 | 生成时间极长 | 建议按模块拆分分批生成 |

---

### PDF 文件 — 标题识别策略

PDF 无法读取样式信息，系统通过以下策略识别标题：

1. **编号前缀**（最可靠）：与 DOCX 相同的编号规则，优先级最高
2. **字体大小**：字号超过正文 15% 以上，识别为一级标题
3. **字号 + 行长**：字号超过正文 5% 且行长不超过 60 字，识别为二级标题

**PDF 最佳实践：**
- 标题字号比正文大至少 2pt，且使用标准数字编号
- 不要将标题做成图片、艺术字或文本框
- 确认 PDF 可检索（用鼠标能框选文字）
- 不含水印遮盖文字

## 生成质量保障机制

### 生成原则

系统采用**严格忠于原文**的扩写策略，核心规则如下：

1. **忠于原意** — 完全保留原文核心含义、业务定位、逻辑框架，不篡改、不删减、不新增无关内容
2. **逐段扩写** — 按原文段落顺序逐段独立扩写，一段原文对应一段扩写内容，保持原有段落结构
3. **文风标准** — 采用政企、园区、数字化平台、项目方案正式书面文风，语言严谨专业
4. **格式规范** — 分段输出，清晰区分每段扩写内容，排版整齐，不改变原有段落顺序
5. **专业适配** — 贴合平台运营、资源汇聚、系统整合、生态建设等政企信息化通用专业语境

### 全局摘要机制

生成开始前，系统会先对整份规范书生成一份约 500 字的项目概述（不消耗用户时间，在后台完成），作为所有章节生成的全局上下文，确保各章节内容的一致性和整体连贯性。

### 空章节处理

原文内容为空的章节（如只有目录条目、无正文段落）：
- **不会调用 LLM**，避免无中生有
- 在最终 Word 文档和目录中**保留标题**，确保文档结构与规范书 TOC 完全一致
- 左侧目录显示为删除线样式，标识为已跳过

## 生成参数设置

### 目标字数

每章节扩写的目标字数，范围 100～10000，默认 500。

| 建议字数 | 适用场景 |
|---------|---------|
| 200～500 | 概述类、目录条目多时快速生成 |
| 500～1000 | 常规技术章节 |
| 1000～2000 | 核心功能章节，需详细展开 |
| 2000 以上 | 重点技术方案，生成时间较长 |

实际字数可能浮动 ±20%，以中文字符为统计单位。目标字数是 AI 的生成指导，不影响对原文结构的忠实程度。

### 字数与页数统计

- 顶部实时显示已生成总字数及预估 Word 页数（基于每页约 800 字估算）
- 各章节完成后，目录条目右侧显示该章节字数

## Word 文档格式说明

### 文档结构

- 不含封面大标题，直接从第一章节内容开始
- 章节标题层级与规范书 TOC 严格对应（最多 4 级）
- 跳过章节（原文为空）仅保留对应层级标题，无正文内容

### 页面与字体规范

| 元素 | 规格 |
|------|------|
| 页面尺寸 | A4（21 × 29.7 cm） |
| 页边距 | 上下左右各 2.5 cm |
| 正文字体 | 宋体，11pt |
| 一级标题 | 16pt，深蓝 #1F3864，段前 12pt |
| 二级标题 | 14pt，蓝色 #2E5496，段前 10pt |
| 三级标题 | 12pt，浅蓝 #2E75B6，段前 8pt |
| 四级标题 | 11pt，深灰 #404040，段前 6pt |
| 代码块 | Courier New 9pt，灰底 #F3F4F6 |
| 表格表头 | 加粗，浅蓝背景 #DBEAFE |

### Markdown 元素转换支持

| Markdown 语法 | Word 元素 |
|--------------|-----------|
| `# ~ ####` 标题 | Heading 1～4 |
| `**加粗**` | 加粗 Run |
| `*斜体*` | 斜体 Run |
| `` `行内代码` `` | Courier New，紫色 |
| ` ```代码块``` ` | 灰底等宽段落 |
| `- / * / +` 列表 | List Bullet |
| `1. 2.` 有序列表 | List Number |
| `\| 表格 \|` | DOCX Table（带表头样式） |

## 本地启动

### 环境要求

- Python 3.10+
- macOS / Linux / Windows（WSL2）

### 一键启动

```bash
cd tech-solution-generator
chmod +x start.sh
./start.sh
```

启动成功后浏览器自动打开 `http://localhost:8000`。

如遇端口占用：

```bash
# 查找并关闭占用进程
lsof -ti :8000 | xargs kill -9
# 或修改 start.sh 中的 PORT 变量
```

### 手动启动

```bash
cd tech-solution-generator
python -m venv .venv
source .venv/bin/activate       # Windows: .venv\Scripts\activate
pip install -r requirements.txt
cd backend
uvicorn main:app --host 0.0.0.0 --port 8000 --reload
```


## 常见问题

**Q：上传后章节识别数量为 0 或很少，怎么处理？**

DOCX：检查是否有目录页（"目录"标题 + 条目列表），或标题是否使用了「标题 1/2/3/4」样式，或是否包含 `一、` `1.` 等标准编号前缀。

PDF：确认 PDF 可检索（鼠标可选中文字），且标题字号明显大于正文。

---

**Q：很多章节显示为"已跳过"（删除线样式），原因是什么？**

被跳过的章节说明在规范书中该章节标题下没有正文内容（只有目录条目，无实质内容段落）。这是正常行为，系统不会对空内容章节进行无中生有的生成。最终 Word 文档中该章节标题仍会保留，确保文档目录结构完整。

若认为章节有内容却被跳过，可能原因是：段落格式导致内容被误判为其他章节的范围，可检查原文该标题下是否确实有正文段落。

---

**Q：生成内容质量不理想，如何提升？**

- 选用能力更强的模型（GPT-4o、Claude 3.5 Sonnet、DeepSeek Chat）
- 适当增加目标字数
- 关键点在于**规范书原文质量**：章节原文越详细（含技术指标、功能描述、约束条件），AI 扩写的参考越充分，输出质量越高
- 原文只有标题而无正文的章节，AI 会自动跳过，避免低质量生成

---

**Q：生成中途断开，已完成内容会丢失吗？**

不会。已完成章节内容保存在服务器内存中，可直接点击「下载 Word」导出已完成部分。刷新页面或重新启动生成才会清空。

---

**Q：豆包 API 调用失败，提示模型不存在？**

豆包（火山引擎）需在控制台创建推理接入点，模型 ID 格式为 `ep-xxxxxxxx-xxxxx`（接入点 ID）。在配置面板「模型」字段填写实际接入点 ID，而非 `doubao-pro-32k`。

---

**Q：如何清除已保存的 API Key？**

删除项目根目录（或 `CONFIG_PATH` 指向目录）下的 `.api_config.json` 文件，重启服务后配置清空。
